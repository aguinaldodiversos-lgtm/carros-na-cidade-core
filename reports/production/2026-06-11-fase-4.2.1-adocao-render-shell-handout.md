Diferença de slugs que confirma a origem dos dados:
- **Fallback hardcoded (antigo):** slugs por cidade, ex. `...-em-atibaia-sp`.
- **CMS adotado (novo):** slugs city-agnostic, ex. `como-comprar-carro-usado-com-seguranca`.

Ver um slug city-agnostic renderizado na página pública prova que o conteúdo veio
do CMS, e não do fallback.

## 8) Validar o público (CMS, sem duplicidade) + revalidar cache

O hub público `/blog/[cidade]` é ISR com `revalidate = 300s` e a busca de posts
tem tag de cache `public-blog`. Após a escrita direta no banco, o cache do Next só
atualiza no próximo TTL (≤ 5 min) **ou** se dispararmos o revalidate.

**Acelerar o revalidate (opcional)** — rodar no Shell do serviço **frontend**
(onde `REVALIDATE_TOKEN` está setado), trocando o domínio se necessário:
```bash
curl -s -X POST https://www.carrosnacidade.com/api/revalidate \
  -H "Authorization: Bearer $REVALIDATE_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"tags":["public-blog"],"paths":["/blog"]}'
# Esperado: {"ok":true,"revalidated":{"paths":["/blog"],"tags":["public-blog"]}}
```

**Provas (HTTP, sem auth)** — ajuste `<cidade>` (ex.: a cidade padrão para a qual
`/blog` redireciona) e o domínio:
```bash
# Hub usa CMS? procure um slug city-agnostic (do CMS) na página renderizada:
curl -s https://www.carrosnacidade.com/blog/<cidade> | grep -o "como-comprar-carro-usado-com-seguranca" | head -1
# Esperado: o slug aparece (origem CMS, não o fallback "...-em-<cidade>")

# Detalhe /blog/<slug> abre a versão CMS (artigo + JSON-LD BlogPosting):
curl -s https://www.carrosnacidade.com/blog/como-comprar-carro-usado-com-seguranca | grep -o "BlogPosting" | head -1
# Esperado: "BlogPosting"
```

**Sem duplicidade:** com 13 posts CMS, o hub passa a montar
featured/popular/trending **somente do CMS** (o fallback hardcoded fica suprimido) —
ver `frontend/app/blog/[cidade]/page.tsx`. Confirme no navegador que cada matéria
aparece uma única vez.

## Conflitos (se aparecer `SKIP-CONFL` no passo 2)

Significa que o slug já existe com `source!='cms'` (linha do motor de SEO). O script
preserva essa linha de propósito. Opções, caso a caso:
- manter o slug do CMS diferente (renomear no dataset), **ou**
- avaliar se a linha SEO deveria virar editorial (decisão manual — não automatizar).

Não force a re-adoção por cima de uma linha SEO. Me acione antes.

## Rollback

A adoção é reversível: **Despublicar/Arquivar** pelo admin (some do público,
permanece editável) ou, em último caso,
`DELETE FROM blog_posts WHERE source='cms' AND tags::text LIKE '%adotado-4.2.1%';`
(remove apenas as linhas adotadas por este lote, identificadas pela tag interna).
